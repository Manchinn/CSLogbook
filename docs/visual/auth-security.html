Section 2

JWT Token Structure

Slim payload design — role-based fields included conditionally

{
  "jti": "unique-id",
  "userId": 1,
  "role": "student|teacher|admin",

  // student-specific
  "studentId": 1,
  "studentCode": "6504062610xxx",

  // teacher-specific
  "teacherId": 1,
  "teacherPermissions": ["teacher:type", "teacher:position:*"],

  // admin-specific
  "isAdmin": true,

  "iat": 1234567890,
  "exp": 1234567890
}

Student Token

Contains studentId + studentCode for data scoping

Teacher Token

Contains teacherId + teacherPermissions[] for sub-role access

Admin Token

Contains isAdmin: true for full system access (teacher:support role)

Section 3

Token Lifecycle & Blacklist

State transitions ของ JWT token ตั้งแต่สร้างจนหมดอายุ

stateDiagram-v2 [*] --> Active : jwt.sign() Active --> Active : API request (valid) Active --> Blacklisted : Logout / Force revoke Active --> Expired : Token TTL exceeded Blacklisted --> [*] : Rejected on next request Expired --> [*] : Rejected on next request note right of Blacklisted Persisted in data/token-blacklist.json Checked on every authenticateToken() call end note

Blacklist Persistence

Revoked tokens are stored in data/token-blacklist.json on the backend. The blacklist is loaded into memory on server start and checked synchronously on every authenticated request via authenticateToken() middleware.

Section 9

Permission Resolution Pipeline

buildPrincipalKeys() flow และ role resolution hierarchy

// Permission Resolution — buildPrincipalKeys()
// authorize() middleware resolves permissions in this order:

function buildPrincipalKeys(user) {
  const keys = [];

  // 1. Base role
  keys.push(user.role);                    // "student" | "teacher" | "admin"

  // 2. Teacher sub-permissions (from JWT)
  if (user.teacherPermissions) {
    user.teacherPermissions.forEach(p => {
      keys.push(p);                        // "teacher:type", "teacher:position:head"
    });
  }

  // 3. Admin shortcut (teacher with support type)
  if (user.isAdmin || user.teacherPermissions?.includes('teacher:support')) {
    keys.push('admin');                    // Full access granted
  }

  return keys;
}

// Resolution hierarchy:
//   admin > teacher:position:* > teacher:type > teacher > student
//
// Wildcard matching:
//   "teacher:position:*" matches "teacher:position:head",
//                                "teacher:position:deputy", etc.
//
// Route-level check:
//   authorize('admin', 'teacher:topic_exam_access')
//   → user must have at least ONE of the listed keys

Section 12

Security Measures

Runtime protection mechanisms

Active Security Measures

Token Expiry Check

Token expiry is verified on every page load via AuthContext. Expired tokens trigger automatic logout and redirect to login.

Automatic Session Expiry

JWT tokens have a configurable TTL. After expiry, the token is rejected by authenticateToken middleware and the user must re-authenticate.

Rate Limiting

Rate limiting is applied to login endpoints and other sensitive routes to prevent brute-force and credential stuffing attacks.

CORS Whitelist

Only whitelisted origins (configured in corsOrigins.js) can make cross-origin requests. Dynamic configuration supports dev and production environments.

KMUTNB SSO Integration

Students authenticate via KMUTNB's official SSO (OAuth). No student passwords stored in CSLogbook database.
